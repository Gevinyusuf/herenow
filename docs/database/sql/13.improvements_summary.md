# 用户权限功能设计审查和改进建议

## 📋 审查范围

- 数据库设计（plans, subscriptions, usages）
- 视图设计（v_user_entitlements）
- 函数设计（check_and_increment_quota）
- 前端实现（useEntitlements hook, 权限控制）
- 后端实现（verify_ai_quota）

## ✅ 设计优点

1. **原子性操作**：使用 `FOR UPDATE` 和 RPC 函数确保并发安全
2. **灵活的 JSONB 设计**：plans 表使用 JSONB 存储限制，易于扩展
3. **视图封装**：v_user_entitlements 统一权限查询接口
4. **依赖注入模式**：后端使用 FastAPI Depends 优雅处理配额验证
5. **前端缓存**：使用 SWR 缓存权限数据，减少请求

## ⚠️ 发现的问题

### 1. **配额重置机制缺失** 🔴 重要

**问题**：
- `current_period_end` 字段存在但未被使用
- 没有定期重置配额的机制
- 用户配额用完后无法自动恢复

**影响**：
- 用户配额用完后永久无法使用
- 无法实现月度/年度配额重置

**建议**：
- 实现配额重置函数（见 `12.quota_reset_function.sql`）
- 设置定时任务（pg_cron 或外部 cron job）
- 在 `check_and_increment_quota` 中检查周期是否过期

### 2. **视图缺少订阅状态检查** 🟡 中等

**问题**：
- `v_user_entitlements` 视图没有检查 `subscriptions.status = 'active'`
- 可能返回已取消或过期的订阅数据

**当前代码**：
```sql
from subscriptions s
join plans p on s.plan_id = p.id
```

**建议**：
```sql
from subscriptions s
join plans p on s.plan_id = p.id
where s.status = 'active'  -- 添加状态检查
```

### 3. **无限额度的处理不一致** 🟡 中等

**问题**：
- 视图返回 `999999` 表示无限
- 函数返回 `true` 表示无限
- 前端判断 `=== -1` 表示无限
- 三种不同的表示方式可能导致混淆

**建议**：
- 统一使用 `-1` 表示无限额度
- 视图中也使用 `-1` 而不是 `999999`

### 4. **用户无订阅时的处理** 🟡 中等

**问题**：
- `v_user_entitlements` 使用 `JOIN`，如果用户没有订阅会返回空
- 前端没有处理用户无订阅的情况
- 可能导致权限检查失败

**建议**：
- 视图改为 `LEFT JOIN`，无订阅时返回默认值
- 前端添加无订阅状态的提示
- 后端 `check_and_increment_quota` 已处理（返回 false），但可以改进错误信息

### 5. **订阅状态处理不完整** 🟡 中等

**问题**：
- `status` 字段有 `active, trailing, canceled` 三种状态
- 但只检查 `active` 状态
- `trailing`（宽限期）状态的处理不明确

**建议**：
- 明确 `trailing` 状态的处理逻辑（是否允许使用功能）
- 在视图中添加状态字段，前端可以显示订阅状态

### 6. **错误信息不够友好** 🟢 轻微

**问题**：
- 配额不足时返回通用错误信息
- 没有告诉用户具体剩余多少配额

**建议**：
- 返回更详细的错误信息，包含剩余配额
- 前端显示更友好的提示

### 7. **缺少配额使用历史记录** 🟢 轻微

**问题**：
- `usages` 表只记录总数，没有历史记录
- 无法追踪配额使用的时间线

**建议**：
- 如果需要审计功能，可以添加 `usage_history` 表
- 记录每次配额使用的详细信息

### 8. **前端权限检查的竞态条件** 🟢 轻微

**问题**：
- 前端权限检查依赖 SWR 缓存
- 如果权限刚更新，可能存在短暂的不一致

**建议**：
- 关键操作在后端再次验证权限
- 前端权限检查主要用于 UI 展示

## 📝 代码修改建议

### 1. 修复 v_user_entitlements 视图

```sql
create or replace view public.v_user_entitlements as
select 
  s.user_id,
  p.name as plan_name,
  p.id as plan_id,
  s.status as subscription_status,  -- 添加状态
  s.current_period_end,  -- 添加周期结束时间
  -- ... 其他字段
from subscriptions s
join plans p on s.plan_id = p.id
left join usages u_ai on s.user_id = u_ai.user_id and u_ai.feature_key = 'quota_ai_generations'
where s.status = 'active'  -- 添加状态过滤
  and (s.current_period_end is null or s.current_period_end > now());  -- 检查周期
```
