# 前端数据与数据库对应关系分析

## 概述
本文档分析 `apps/frontend/app/(main)/create/event/page.tsx` 中的前端数据结构与 `docs/database/sql/04_events_v1.sql` 数据库表结构的对应关系。

## 前端 EventData 接口

```typescript
interface EventData {
  eventName: string;                    // 活动名称
  coverImage: string;                   // 封面图片 URL
  startDate: string;                    // 开始日期 (YYYY-MM-DD)
  startTime: string;                    // 开始时间 (HH:MM)
  endDate: string;                      // 结束日期 (YYYY-MM-DD)
  endTime: string;                      // 结束时间 (HH:MM)
  location: string;                     // 地点名称
  description: string;                  // 描述 (HTML)
  tickets: Ticket[];                    // 票务配置数组
  questions: Question[];                // 报名问题数组
  host: { id: string; name: string; type: string; icon: any };
  coHosts: any[];                       // 联合主办方
  selectedTheme: Theme;                 // 主题配置
  selectedEffect: string;               // 特效
  selectedTimezone: { id: string; city: string; offset: string; currentTime: string };
  visibility: 'public' | 'private';    // 可见性
  isVirtual: boolean;                   // 是否虚拟活动
  meetingLink: string;                  // 会议链接
  requireApproval: boolean;             // 是否需要审核
  isLocationPublic?: boolean;           // 地点是否公开（前端有但未在 EventData 中定义）
}
```

## 数据库表结构 (events_v1)

```sql
CREATE TABLE events_v1 (
    id UUID PRIMARY KEY,
    title TEXT NOT NULL,                -- 活动名称
    slug TEXT UNIQUE NOT NULL,          -- URL 友好标识
    description TEXT,                    -- 描述 (HTML)
    cover_image_url TEXT,                -- 封面图片
    start_at TIMESTAMPTZ NOT NULL,      -- 开始时间
    end_at TIMESTAMPTZ NOT NULL,        -- 结束时间
    timezone TEXT NOT NULL,              -- 时区
    location_info JSONB,                 -- 地点信息
    visibility TEXT DEFAULT 'public',    -- 可见性
    require_approval BOOLEAN DEFAULT FALSE,
    host_id UUID REFERENCES profiles(id),
    style_config JSONB,                  -- 主题和特效配置
    registration_fields JSONB,           -- 报名表单字段
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 字段对应关系

### ✅ 直接对应（无需转换）

| 前端字段 | 数据库字段 | 说明 |
|---------|-----------|------|
| `eventName` | `title` | 活动名称 |
| `coverImage` | `cover_image_url` | 封面图片 URL |
| `description` | `description` | HTML 描述 |
| `visibility` | `visibility` | 'public' 或 'private' |
| `requireApproval` | `require_approval` | 是否需要审核 |
| `host.id` | `host_id` | 主办方 ID |

### ⚠️ 需要转换的字段

#### 1. 时间字段组合

**前端：**
- `startDate`: "2025-11-19"
- `startTime`: "20:00"
- `endDate`: "2025-11-19"
- `endTime`: "21:00"
- `selectedTimezone.id`: "America/New_York"

**数据库：**
- `start_at`: TIMESTAMPTZ (需要组合日期和时间)
- `end_at`: TIMESTAMPTZ (需要组合日期和时间)
- `timezone`: TEXT (需要提取 timezone id)

**转换逻辑：**
```typescript
// 需要在前端或后端 API 中转换
const startAt = new Date(`${startDate}T${startTime}:00`).toISOString();
const endAt = new Date(`${endDate}T${endTime}:00`).toISOString();
const timezone = selectedTimezone.id; // 例如 "America/New_York"
```

#### 2. 地点信息 (location_info JSONB)

**前端：**
- `isVirtual`: boolean
- `location`: string (离线活动的地点名称)
- `meetingLink`: string (虚拟活动的链接)
- `isLocationPublic`: boolean (地点是否公开，前端有但未在 EventData 中定义)

**数据库：**
- `location_info`: JSONB

**转换逻辑：**
```typescript
const locationInfo = isVirtual
  ? {
      type: "virtual",
      link: meetingLink
    }
  : {
      type: "offline",
      name: location,
      isPublic: isLocationPublic || false
      // 注意：如果需要经纬度，需要从 Google Places API 获取
      // lat: number,
      // lng: number
    };
```

#### 3. 主题和特效配置 (style_config JSONB)

**前端：**
- `selectedTheme`: Theme 对象（包含 id, name, category, bg, contentBg, text, button, preview）
- `selectedEffect`: string (例如 "sparkles", "none")

**数据库：**
- `style_config`: JSONB

**转换逻辑：**
```typescript
const styleConfig = {
  themeId: selectedTheme.id,
  effect: selectedEffect,
  colors: {
    bg: selectedTheme.bg,
    contentBg: selectedTheme.contentBg,
    text: selectedTheme.text,
    button: selectedTheme.button
  },
  preview: selectedTheme.preview
};
```

#### 4. 报名表单字段 (registration_fields JSONB)

**前端：**
- `questions`: Question[] 数组

```typescript
interface Question {
  id: string;        // 例如 'name', 'email', 'q_1234567890'
  label: string;     // 问题标签
  required: boolean; // 是否必填
  fixed: boolean;    // 是否固定字段（name, email 等）
  saveToTemplate: boolean; // 是否保存到模板
}
```

**数据库：**
- `registration_fields`: JSONB

**转换逻辑：**
```typescript
const registrationFields = questions.map(q => ({
  id: q.id,
  label: q.label,
  required: q.required,
  fixed: q.fixed
  // saveToTemplate 是前端模板功能，不需要存到数据库
}));
```

#### 5. Slug 生成

**前端：** 无此字段

**数据库：** `slug` TEXT UNIQUE NOT NULL

**转换逻辑：**
```typescript
// 需要后端生成或前端生成
const slug = generateSlug(eventName); 
// 例如: "Product Launch 2025" -> "product-launch-2025"
// 需要确保唯一性，可能需要添加随机后缀或时间戳
```

### ❌ 前端有但数据库表中没有的字段

#### 1. Tickets（票务配置）

**前端：**
```typescript
interface Ticket {
  id: number;
  name: string;
  type: 'free' | 'paid';
  price: string;
  quantity: string;
  requireApproval: boolean;
  saveToTemplate: boolean;
}
```

**数据库：** `events_v1` 表中没有票务相关字段

**建议：**
- 方案1：将 tickets 存入 `style_config` 或新增 `ticket_config` JSONB 字段
- 方案2：创建独立的 `event_tickets` 表（推荐用于复杂票务系统）

**如果使用 JSONB 存储：**
```typescript
const ticketConfig = {
  tickets: tickets.map(t => ({
    id: t.id,
    name: t.name,
    type: t.type,
    price: t.type === 'paid' ? parseFloat(t.price) : 0,
    quantity: parseInt(t.quantity) || null, // null 表示无限制
    requireApproval: t.requireApproval
  }))
};
```

#### 2. CoHosts（联合主办方）

**前端：** `coHosts: any[]`

**数据库：** `events_v1` 表中没有联合主办方字段

**建议：**
- 方案1：创建独立的 `event_co_hosts` 表（推荐）
- 方案2：将 coHosts 存入 JSONB 字段（简单场景）

## 需要处理的问题

### 1. ⚠️ 时间处理
- 前端需要将 `startDate + startTime` 和 `endDate + endTime` 组合成 ISO 8601 格式
- 需要考虑时区转换，确保时间正确

### 2. ⚠️ Slug 生成
- 需要从 `eventName` 生成唯一的 slug
- 需要处理重复情况（添加后缀或时间戳）

### 3. ⚠️ Tickets 存储
- `events_v1` 表中没有票务字段
- 需要决定是使用 JSONB 还是独立表

### 4. ⚠️ CoHosts 存储
- `events_v1` 表中没有联合主办方字段
- 需要决定存储方式

### 5. ⚠️ isLocationPublic 字段
- 前端有 `isLocationPublic` 状态，但未在 `EventData` 接口中定义
- 需要确认是否要存储到数据库

### 6. ⚠️ 地点经纬度
- 如果使用 Google Places API，可以获取经纬度
- 建议在 `location_info` 中存储 `lat` 和 `lng`

## 建议的 API 转换函数

```typescript
// lib/api/eventTransform.ts
export function transformEventDataForAPI(eventData: EventData): CreateEventPayload {
  // 组合时间
  const startAt = new Date(`${eventData.startDate}T${eventData.startTime}:00`).toISOString();
  const endAt = new Date(`${eventData.endDate}T${eventData.endTime}:00`).toISOString();
  
  // 生成 slug（简单版本，实际需要确保唯一性）
  const slug = eventData.eventName
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/^-|-$/g, '') + '-' + Date.now().toString(36);
  
  return {
    title: eventData.eventName,
    slug,
    cover_image_url: eventData.coverImage,
    description: eventData.description,
    start_at: startAt,
    end_at: endAt,
    timezone: eventData.selectedTimezone.id,
    location_info: eventData.isVirtual
      ? { type: "virtual", link: eventData.meetingLink }
      : { 
          type: "offline", 
          name: eventData.location,
          isPublic: eventData.isLocationPublic || false
        },
    visibility: eventData.visibility,
    require_approval: eventData.requireApproval,
    host_id: eventData.host.id,
    style_config: {
      themeId: eventData.selectedTheme.id,
      effect: eventData.selectedEffect,
      colors: {
        bg: eventData.selectedTheme.bg,
        contentBg: eventData.selectedTheme.contentBg,
        text: eventData.selectedTheme.text,
        button: eventData.selectedTheme.button
      }
    },
    registration_fields: eventData.questions.map(q => ({
      id: q.id,
      label: q.label,
      required: q.required,
      fixed: q.fixed
    }))
  };
}
```

## 总结

### ✅ 已对应字段（7个）
- eventName → title
- coverImage → cover_image_url
- description → description
- visibility → visibility
- requireApproval → require_approval
- host.id → host_id
- (时间字段需要组合但逻辑清晰)

### ⚠️ 需要转换字段（5个）
- startDate + startTime → start_at
- endDate + endTime → end_at
- selectedTimezone.id → timezone
- location/isVirtual/meetingLink → location_info (JSONB)
- selectedTheme + selectedEffect → style_config (JSONB)
- questions → registration_fields (JSONB)

### ❌ 缺失字段（3个）
- tickets → 需要决定存储方式
- coHosts → 需要决定存储方式
- slug → 需要生成

### 📝 建议
1. 在 API 层添加数据转换函数
2. 考虑将 tickets 和 coHosts 存储到 JSONB 字段或独立表
3. 实现 slug 生成逻辑（前端或后端）
4. 确认 isLocationPublic 是否需要存储
